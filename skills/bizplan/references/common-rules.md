# 공통 규칙: 지원사업 성격 파악 (전 모드 공용)

모든 bizplan 모드는 작업 시작 전 아래를 확인한다. 이 파일이 단일 기준이며, 각 서브스킬에는 한 줄 요약만 둔다. 정책·수치가 바뀌면 **이 파일만 수정**하면 된다.

## 파악할 항목

1. **주관기관과 사업명**: 예비창업패키지 / 청년창업사관학교(청창사) / U300 / 로컬크리에이터 / 수출바우처 / 기타
2. **평가 기준·배점**: 공식 배점표가 있으면 반드시 첨부를 요청한다 — 배점 항목에 본문을 매핑하는 것이 점수의 지름길이다.
3. **지원 규모·기간**: 금액·기간은 해당 공고 기준으로 조정한다. 원본의 예창패 기준 수치(1단계 2천만 + 2단계 4천만, 8개월)는 기본값 예시로만 쓴다.
4. **양식 구조**: 예창패 PSST형 / 청창사 기술성·시장성·사업성형 / U300형 / 로컬크리에이터 지역성형
5. **심사 방식**: 서면 / 발표. 발표 심사가 있으면 초안 완성 후 `bizplan-pitch`로 연결한다.

## 정부지원사업이 아닌 경우

투자 유치 IR, 민간 공모전, 정책자금·대출 심사, 사내벤처, B2B 제안 등이면 해당 심사·평가 자료(IR 가이드라인, 공모 요강, 심사 항목표) 첨부를 요청한다. 자료가 없으면 유형별 부각 포인트를 먼저 안내한다:

- **IR** = 팀·시장·트랙션·유닛 이코노믹스·투자금 사용처(라운드·희망 밸류·엑싯 전략 포함)
- **공모전·경진대회** = 주최 취지 부합·창의성·발표 전달력 가중(재무·양산 계획 비중↓, 통상 5~10페이지 — **공공 주최는 지식재산 귀속·공공 활용 동의 조항이 있는 경우가 많으니 확인하도록 고지**)
- **정책자금·대출** = 상환 능력·재무 건전성
- **B2B 제안** = 고객사 문제 해결과 ROI

이후 각 모드의 출력 형식을 그 유형에 맞게 조정한다.

## 제출 양식 핏 (양식 파일 제공 시)

사용자가 제출 양식(공고 양식 hwp/hwpx/docx/pdf 파일, 목차 캡처, 항목표 텍스트)을 제공하면 **양식이 스킬 기본 구조보다 우선**한다. 모든 모드가 다음을 따른다:

1. **목차·순서 준수**: 산출물의 목차·항목명·항목 번호·순서를 양식 그대로 따르고, 각 산출물 헤더에 양식 항목 번호(예: 1-1, 2-2)를 표기한다.
2. **항목별 지시문·분량 준수**: 양식에 적힌 항목별 작성 지침·분량 제한(페이지·글자 수)을 지킨다. 양식의 안내 문구(예: 예창패 파란 안내 박스)는 제출 전 삭제 대상으로 표시한다.
3. **차이 처리**: 스킬 기본 구조에는 있으나 양식에 없는 내용은 가장 가까운 양식 항목에 병합하고, 양식에는 있으나 초안에 없는 항목은 "**[작성 필요: 항목명]**"으로 남겨 누락을 드러낸다.
4. **읽기 실패 폴백**: 양식 파일을 직접 읽을 수 없는 형식(hwp 등)이면 목차·항목 텍스트를 채팅에 붙여넣어 달라고 요청한다.

양식을 받기 전에 이미 만든 산출물이 있으면, 양식 확보 시점에 그 목차 기준으로 재배치한다.

## 연혁·보유 자산·활동 로그 반영 (제공 시)

사용자(팀)가 실제 이력·자산 자료를 제공하면 일반적인 서술 대신 **그 자료의 사실(날짜·수치·성과)을 근거로 작성**한다. 증빙 없는 일반론보다 실제 이력이 담긴 계획서가 심사에서 훨씬 유리하다.

- **대상 자료**: 연혁(창업·수상·제품 출시·투자 유치·계약·팀 합류 등 시간순 이력) / 팀 어셋(경력·포트폴리오·특허·인증·시제품) / 활동 로그(개발 일지·회의록·실험·실측 기록·고객 인터뷰·런칭 기록) / 증빙(MOU·LOI·매출 자료)
- **배치**: 연혁·팀 어셋은 팀 구성(대표자 이력·팀 연혁)에, 개발·실험 로그는 실현 가능성(개발 경과·검증 이력)에, 고객 인터뷰·매출 자료는 문제 인식·성장 전략에 우선 반영한다.
- **표기**: 출처를 "(팀 제공 자료)"로 표기하고, 제공 자료와 모순되는 내용을 만들지 않는다. 개인정보 이니셜 관례는 그대로 적용한다.
- 세션에서 한 번(라우터 후속 질문 또는 각 모드 시작 시) 보유 여부를 확인하고, 이미 받은 자료는 다시 요청하지 않는다.

## 보강 제안·자료 요청 (산출물 공통)

본문을 생성하는 모든 모드(item·problem·research·solution·growth·team·summary·quick·export)는 산출물 말미에 **"보강 제안"** 블록을 붙인다:

1. **있으면 좋을 내용**: 이 섹션을 더 강하게 만들 추가 내용 제안(예: 연혁, 실측 데이터, 고객 인용, 파일럿 결과, 재구매율) — 각 항목에 심사 관점에서 왜 유리한지 한 줄 이유를 단다.
2. **요청 자료**: 사용자에게 요청할 자료 목록(공고문·배점표·제출 양식, 연혁·어셋·활동 로그, 구매의향서·시험성적서 등 증빙)을 우선순위와 함께 제시하고, 받으면 어느 항목에 반영할지 명시한다.
3. `bizplan-review`는 별도 블록 대신 기존 '약점 및 개선점'·'필수 수정'에 요청 자료를 연결해 제시한다.

## 확인 방법

사용자가 모르면 공고문 파일을 요청하거나 웹 검색으로 파악한다. 대화에 이미 정보가 있으면 다시 묻지 않는다.

## U300 등 학생 트랙 — 깊이 인터뷰

학생 팀은 원가 데이터·시제품·매출 실적 같은 증빙 자산이 없는 경우가 많다. 예창패 수준의 깊이를 일괄 적용하지 말고, 아래 3~4가지로 **무엇을 갖고 있는지 항목별로** 먼저 확인한다.

① **원가·단가 데이터** — 견적서·BOM·재료비 계산이 있는가
② **시제품·실측·증빙** — 시제품(사진), 성능 실측치, 시험 결과, 1차 조사(설문·인터뷰)가 있는가
③ **매출·계약 실적** — 판매 이력, 구매의향서(LOI), PoC 협의가 있는가
④ (재무 추정을 쓸 계획이면) **추정 근거** — 고객 수·단가·전환율 가정을 세워 봤는가

**있는 항목은 예창패 수준 깊이로 쓰고, 없는 항목은 "산정 예정(근거·확보 시점 명시)" 폴백으로 처리한다**(`bizplan-solution` B-1 원칙). 전체를 일괄 간략화하지 않는다 — 항목별 판단이 원칙이다. 이 확인 결과는 이후 모드(`bizplan-solution`·`bizplan-research`·`bizplan-review`)가 재사용한다.

## 출력 형식 (세션 공통)

세션에서 정한 형식(채팅/.md/.docx)을 모든 모드가 따른다. 파일 모드면 결과를 파일에 누적하고 채팅에는 요약만 보여준다. 정해진 것이 없으면 채팅 출력을 기본으로 하되, 긴 문서면 파일 저장을 한 번 제안한다. 상세 절차(환경별 파일 처리·.docx 변환)는 `bizplan` SKILL.md의 '출력 형식 (세션 공통)' 절을 참조한다.

## 개인정보·이니셜 관례

학교명·직장명 등 개인정보는 기재하지 않는다. 부득이하면 'S대'·'K사'처럼 이니셜로 처리한다.
